# 2026-07-02 · Guidance：条件生成不是“加提示词”，而是在改写 score 方向

> 阶段 B · 第 4 篇 ｜ 标签：diffusion, guidance, classifier guidance, classifier-free guidance, CFG, conditional generation
> 读法：接着 B3。今天不要先背 CFG 公式，先抓住一句话：条件生成就是让反向采样的每一步，不只“更像数据”，还要“更像条件”。

## ⏮ 昨日回顾（30 秒）

B3 把 DDPM 和 DDIM 收进了 score / SDE 地图里。网络预测噪声 `ε_θ(x_t,t)`，本质上等价于预测 score：

```text
s_θ(x_t,t) ≈ ∇_{x_t} log p_t(x_t)
```

这里 `p_t(x_t)` 是噪声水平 `t` 下的图像分布。score 指向“让当前样本更像真实数据”的方向。

今天往前走一步：如果我们不只想生成“像真实图像”的样本，还想生成“满足条件 `y` 的图像”，比如文本、类别、RGB 条件、边缘图、法向图、偏振先验，那么采样方向就应该从：

```text
∇ log p_t(x)
```

变成：

```text
∇ log p_t(x | y)
```

这就是 guidance 的核心。

## 🎯 主菜：条件生成与 Guidance

### 台阶 1：条件生成到底在改什么

先定义符号：

- `x_t`：第 `t` 个噪声水平下的带噪图像；
- `y`：条件，可以是类别 label、文本 prompt、RGB 图、深度图、边缘图等；
- `p_t(x_t)`：不看条件时，当前噪声图像的分布；
- `p_t(x_t | y)`：给定条件 `y` 后，当前噪声图像的条件分布；
- `∇_{x_t} log p_t(x_t)`：无条件 score；
- `∇_{x_t} log p_t(x_t | y)`：条件 score。

无条件 diffusion 的采样目标是：从纯噪声走回“像真实数据”的区域。条件 diffusion 的采样目标更窄：从纯噪声走回“既像真实数据，又符合条件 `y`”的区域。

所以 guidance 不是神秘插件，它只是回答一个问题：

> 反向去噪时，每一步应该怎样调整方向，才能把样本推向满足条件的高概率区域？

### 台阶 2：用 Bayes 公式拆条件 score

从 Bayes 公式开始：

```text
p_t(x | y) = p_t(y | x) p_t(x) / p(y)
```

两边取 log：

```text
log p_t(x | y) = log p_t(y | x) + log p_t(x) - log p(y)
```

现在对 `x` 求梯度。注意 `p(y)` 和 `x` 无关，所以：

```text
∇_x log p(y) = 0
```

于是：

```text
∇_x log p_t(x | y)
= ∇_x log p_t(x) + ∇_x log p_t(y | x)
```

这就是 classifier guidance 的数学骨架。逐项翻译：

- `∇_x log p_t(x)`：让样本更像真实图像；
- `∇_x log p_t(y | x)`：让样本更像条件 `y`；
- 二者相加：既真实，又符合条件。

所以如果你手上有一个分类器 `p_φ(y | x_t,t)`，它能判断带噪图 `x_t` 属于类别 `y` 的概率，那么就可以用它的梯度修正 diffusion 的 score：

```text
s_guided(x_t,t,y)
= s_θ(x_t,t) + w · ∇_{x_t} log p_φ(y | x_t,t)
```

这里 `w` 是 guidance scale，控制条件力量有多强。

### 台阶 3：为什么要乘一个 `w`

理论上，Bayes 拆出来的系数是 `1`。工程上常写成：

```text
s_guided = s_uncond + w · s_classifier
```

因为真实模型不是完美的：

- diffusion score 可能偏保守；
- 分类器梯度可能太弱；
- 条件越强，样本越贴合 `y`，但也越容易失真或多样性下降。

所以 `w` 是一个质量-多样性旋钮：

- `w = 0`：退回无条件生成；
- `w` 较小：条件影响轻，图像自然但可能不太听话；
- `w` 较大：更听条件，但可能过锐、过饱和、模式变窄。

这解释了为什么 Stable Diffusion 里 CFG scale 太大时，图像会变得“硬”、细节怪、颜色炸：条件方向被放得太猛，反向轨迹被拉偏了。

### 台阶 4：classifier guidance 的代价

Classifier guidance 很直观，但有两个现实问题。

第一，要额外训练一个“能看懂带噪图”的分类器：

```text
p_φ(y | x_t,t)
```

普通分类器只看干净图 `x_0`，而 guidance 需要它在每个噪声水平 `t` 下都能给出可靠梯度。高噪声时图像像雪花，分类器还要判断类别，这并不容易。

第二，条件不一定是离散类别。文本 prompt、RGB 条件图、边缘图、多模态控制，都很难直接写成一个标准分类器。

于是出现了更常用的 **classifier-free guidance**。

### 台阶 5：Classifier-free guidance 的主意

Classifier-free guidance（CFG）不训练额外分类器，而是让同一个 diffusion model 同时学两种预测：

```text
ε_θ(x_t,t,y)      条件预测
ε_θ(x_t,t,∅)      无条件预测
```

这里 `∅` 表示“没有条件”。训练时会随机把一部分条件 `y` 丢掉，让模型学会：

- 条件在时：按条件生成；
- 条件被 drop 时：做无条件生成。

采样时，把两次预测组合起来：

```text
ε_cfg = ε_uncond + w · (ε_cond - ε_uncond)
```

其中：

- `ε_cond = ε_θ(x_t,t,y)`；
- `ε_uncond = ε_θ(x_t,t,∅)`；
- `ε_cond - ε_uncond`：条件带来的方向差；
- `w`：把这份条件差放大多少。

这条式子就是 CFG 最常见的工程形式。

### 台阶 6：为什么 `ε_cond - ε_uncond` 能当作条件方向

从 score 角度看，无条件模型近似：

```text
s_uncond ≈ ∇_x log p_t(x)
```

条件模型近似：

```text
s_cond ≈ ∇_x log p_t(x | y)
```

而台阶 2 已经推过：

```text
∇_x log p_t(x | y)
= ∇_x log p_t(x) + ∇_x log p_t(y | x)
```

所以：

```text
s_cond - s_uncond
≈ ∇_x log p_t(y | x)
```

也就是说，条件 score 和无条件 score 的差，近似就是“让样本更符合条件”的那部分方向。

CFG 只是把这个差值放大：

```text
s_cfg = s_uncond + w · (s_cond - s_uncond)
```

由于很多实现让模型输出 `ε` 而不是 score，公式会写成 `ε` 版本。符号方向会因参数化和采样公式略有差异，但核心不变：

> 用“条件预测”和“无条件预测”的差，估计条件想把样本往哪里拉。

### 台阶 7：CFG scale 的真实含义

把公式改写一下：

```text
ε_cfg = ε_uncond + w(ε_cond - ε_uncond)
      = w ε_cond + (1-w) ε_uncond
```

当 `w=1`：

```text
ε_cfg = ε_cond
```

就是普通条件模型。

当 `w>1`，模型不是简单“使用条件”，而是在做外推：沿着从无条件预测指向条件预测的方向继续往前推。这就是为什么它会更听 prompt，也更容易过头。

工程直觉：

- 小 `w`：自然、多样，但条件弱；
- 中等 `w`：常用甜点区；
- 过大 `w`：条件强但容易 artifact、过饱和、结构崩。

### 最小可运行代码：CFG 组合

```python
import torch


@torch.no_grad()
def cfg_eps(model, x_t, t, cond, null_cond, guidance_scale=7.5):
    """
    model: predicts epsilon
    x_t: noisy sample [B,C,H,W]
    t: timestep tensor
    cond: real condition, e.g. text embedding or image condition
    null_cond: dropped/empty condition
    guidance_scale: CFG scale w
    """
    eps_cond = model(x_t, t, cond)
    eps_uncond = model(x_t, t, null_cond)
    return eps_uncond + guidance_scale * (eps_cond - eps_uncond)


@torch.no_grad()
def cfg_batch_trick(model, x_t, t, cond, null_cond, guidance_scale=7.5):
    """
    Same computation, but concatenate batch to run the model once.
    """
    x_in = torch.cat([x_t, x_t], dim=0)
    t_in = torch.cat([t, t], dim=0)
    cond_in = torch.cat([null_cond, cond], dim=0)
    eps_uncond, eps_cond = model(x_in, t_in, cond_in).chunk(2, dim=0)
    return eps_uncond + guidance_scale * (eps_cond - eps_uncond)
```

训练时的 condition dropout 也很简单：

```python
def drop_condition(cond, null_cond, drop_prob=0.1):
    keep = torch.rand(cond.shape[0], device=cond.device) > drop_prob
    while keep.ndim < cond.ndim:
        keep = keep.view(*keep.shape, 1)
    return torch.where(keep, cond, null_cond)
```

### 🔗 与你 CVPR 工作的连接

如果你的目标是“给 RGB 图生成偏振表征 / DoLP / AoLP / Polarization Encoding”，那条件不是一句文本，而是视觉条件：

- RGB image；
- depth / normal / edge；
- DoLP / AoLP 的物理先验；
- 可靠性 mask 或 uncertainty map。

CFG 给你一个非常适合写进论文的方法论语言：

> 条件控制不是把条件粗暴拼进网络，而是在反向扩散的每个噪声水平上，估计“满足观测条件的 score 方向”，并用 guidance scale 调节 fidelity 与 diversity 的权衡。

对应实验可以这样设计：

- 固定 sampler 和 seed，扫 `guidance_scale`；
- 看偏振指标、视觉质量、多样性如何变化；
- 如果 scale 太大导致 DoLP 过平滑或 AoLP 结构异常，就可以解释为条件方向过强，破坏了数据先验。

### 一句话总结

> Guidance 的本质，是把无条件 score `∇ log p_t(x)` 改成条件 score `∇ log p_t(x|y)`。Classifier guidance 用外部分类器给出条件梯度；classifier-free guidance 用同一个模型的“条件预测 - 无条件预测”估计条件方向，再用 CFG scale 调节听话程度和生成质量的权衡。

## 🔧 配菜·面试考点

**Q：CFG 为什么不需要额外分类器？**

**A：** 因为训练时通过 condition dropout，让同一个 diffusion model 同时学会条件预测 `ε_θ(x_t,t,y)` 和无条件预测 `ε_θ(x_t,t,∅)`。从 score 角度看，二者分别近似 `∇ log p_t(x|y)` 和 `∇ log p_t(x)`，它们的差近似 `∇ log p_t(y|x)`，也就是条件想施加的方向。采样时用 `ε_uncond + w(ε_cond - ε_uncond)` 放大这部分方向，就能增强条件控制。

追问预备：`w=1` 和 `w>1` 的区别；为什么 CFG scale 过大容易 artifact；image condition 和 text condition 在结构上有什么不同。

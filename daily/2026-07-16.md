## 🎯 主菜：把 diffusion 训练成一条可靠工程流水线

### 台阶 1：先固定训练里真正发生的四件事

一条最基本的 diffusion 训练迭代只有四步：

1. 从数据集中取干净样本 `x_0`；
2. 随机采样 timestep `t` 和高斯噪声 `ε`；
3. 用闭式公式直接得到 `x_t`；
4. 让网络从 `x_t, t, condition` 预测某个目标，并计算损失。

定义：

```text
α_t = sqrt(α_bar_t)
σ_t = sqrt(1 - α_bar_t)
ε ~ N(0, I)
```

则前向加噪为：

```text
x_t = α_t x_0 + σ_t ε
```

这里 `α_bar_t` 是累计信号保留率。注意训练时不需要真的从 `x_0` 一步步加噪到 `x_t`；闭式公式让我们一次就能采到任意噪声水平。

如果采用最常见的 `ε-prediction`，目标就是：

```text
L_ε = E || ε - ε_θ(x_t, t, c) ||²
```

`c` 是可选条件，例如 RGB、文本、DoLP、AoLP 或其他物理表征。

### 台阶 2：noise schedule 决定模型每天在练什么题

通常先定义每一步新增噪声量 `β_t`：

```text
α_step,t = 1 - β_t
α_bar_t = product_{s=1}^{t} α_step,s
```

于是：

- `α_bar_t` 大：`x_t` 里原信号多，是低噪声题；
- `α_bar_t` 小：`x_t` 接近纯噪声，是高噪声题。

常见 linear schedule 让 `β_t` 线性增加。它简单，但“`β_t` 线性”并不等于“学习难度线性”：累乘后的 `α_bar_t` 可能在某段下降太快，使大量 timestep 都几乎看不到信号。

cosine schedule 更直接地设计累计信号曲线。一个常见形式是：

```text
f(t) = cos²(((t / T + s) / (1 + s)) · π/2)
α_bar_t = f(t) / f(0)
```

其中：

- `T` 是总 timestep 数；
- `s` 是很小的偏移，避免起点附近变化过于尖锐；
- `t = 0, ..., T`。

它的直觉不是“余弦天然更高级”，而是让信号在训练时间轴上消失得更平滑，给中等噪声区间留下更多有效学习机会。

判断 schedule 时最好看 signal-to-noise ratio：

```text
SNR(t) = α_t² / σ_t² = α_bar_t / (1 - α_bar_t)
```

schedule 的实质，就是在安排训练样本覆盖怎样的 SNR 分布。

### 台阶 3：为什么同一个 MSE 在不同 timestep 上并不一样难

假设我们均匀采样 `t`。这只保证每个离散 timestep 出现概率一样，并不保证每个 timestep 对梯度的贡献同样合理。

在高 SNR 区：

```text
x_t ≈ x_0
```

网络很容易看清结构，但预测一个很小的噪声扰动可能要求精细拟合。

在低 SNR 区：

```text
x_t ≈ ε
```

原始结构几乎被淹没，条件信息和数据先验更重要。

所以训练稳定性至少要同时检查：

- timestep 是否均匀采样；
- loss 是否被少数 SNR 区域支配；
- 条件在高噪声处是否仍能影响网络；
- 不同目标通道的尺度是否一致。

对 dense physical modality，最后一点尤其重要。若 DoLP、法向量编码、RGB 的数值范围不同，不先归一化，所谓“diffusion loss”可能只是在优先拟合幅值最大的通道。

### 台阶 4：从 `ε-prediction` 完整推到 `v-prediction`

`ε-prediction` 让网络预测加入的噪声。另一种常用参数化是 `v-prediction`：

```text
v_t = α_t ε - σ_t x_0
```

为什么这个定义有用？先不要背结论，我们把它和 `x_t` 联立：

```text
x_t = α_t x_0 + σ_t ε
v_t = -σ_t x_0 + α_t ε
```

把第一式乘 `α_t`，第二式乘 `σ_t` 后相减：

```text
α_t x_t - σ_t v_t
= α_t(α_t x_0 + σ_t ε) - σ_t(-σ_t x_0 + α_t ε)
= α_t² x_0 + α_tσ_t ε + σ_t² x_0 - α_tσ_t ε
= (α_t² + σ_t²)x_0
```

因为：

```text
α_t² + σ_t² = α_bar_t + (1 - α_bar_t) = 1
```

所以：

```text
x_0 = α_t x_t - σ_t v_t
```

同理，把第一式乘 `σ_t`，第二式乘 `α_t` 后相加：

```text
σ_t x_t + α_t v_t
= σ_t(α_t x_0 + σ_t ε) + α_t(-σ_t x_0 + α_t ε)
= (σ_t² + α_t²)ε
= ε
```

得到：

```text
ε = σ_t x_t + α_t v_t
```

这说明网络只要预测 `v_t`，采样时就能同时恢复 `x_0` 和 `ε`。

它的直觉是：`v` 把“数据方向”和“噪声方向”按当前噪声水平旋转到一个统一坐标里。在高、低 SNR 两端，它的目标尺度通常比直接预测某一端更平衡。因此一些 latent diffusion 系统更偏好 `v-prediction`。

但要注意：prediction type 是训练—推理契约。训练时学 `v`，scheduler 却把输出当 `ε`，结果不会只是略差，而是每一步更新公式都解释错了。

### 台阶 5：EMA 为什么常常比最后一份训练权重更好

优化器更新的参数记为 `θ_k`。即使整体在收敛，mini-batch 噪声也会让它在好解附近来回晃动。

Exponential Moving Average 保存一份不参与反向传播的平滑参数：

```text
θ_EMA,k = decay · θ_EMA,k-1 + (1 - decay) · θ_k
```

例如 `decay = 0.9999` 时，新参数每次只占很小比例。展开递推可以看到，EMA 是许多历史参数的指数加权平均，越新的权重越大，越旧的权重影响指数衰减。

它解决的不是“让训练 loss 更低”，而是：

- 减少单个 batch 带来的参数抖动；
- 让采样质量更稳定；
- 避免恰好保存到一次不理想的瞬时更新。

工程上必须区分三份状态：

```text
train weights    用于反向传播
EMA weights      用于验证、采样和最终发布
optimizer state  用于断点续训
```

只保存 EMA 可以推理，但不能完整续训；只保存训练权重可以续训，却可能复现不了报告里的最佳采样质量。

### 台阶 6：mixed precision 省下的是什么，风险又在哪里

Mixed precision 通常让大部分矩阵运算使用 `float16` 或 `bfloat16`，把对数值敏感的部分保留在 `float32`。

收益来自：

- 激活占用更少显存；
- Tensor Core 吞吐更高；
- 同样显存可提高 batch size 或分辨率。

`float16` 的风险是可表示范围较窄。梯度太小时会下溢成 0，所以常配合 loss scaling：

```text
scaled_loss = scale · loss
scaled_grad = scale · grad
optimizer.step 前再除以 scale
```

PyTorch 的 `GradScaler` 会动态调整 `scale`：检测到 `inf/NaN` 时跳过这次更新并降低尺度，稳定时再逐渐提高。

`bfloat16` 的指数范围接近 `float32`，通常不需要 gradient scaling，但有效尾数更短。硬件支持时，它常比 `float16` 更省心。

需要重点保留或检查 `float32` 的位置包括：

- schedule 的累计乘积与 SNR 计算；
- 很小方差的除法、开方和对数；
- loss reduction；
- EMA master weights；
- 自定义物理约束里接近奇点的运算。

### 台阶 7：gradient accumulation 不是免费的大 batch

单卡放不下目标 batch 时，可以累积 `K` 个 micro-batch：

```text
effective batch = per_gpu_batch × world_size × accumulation_steps
```

每个 micro-batch 的 loss 要除以 `K`，累积 `K` 次后再 `optimizer.step()`。否则梯度会被放大 `K` 倍。

它在数学上接近大 batch，但工程上仍有差异：

- BatchNorm 的统计量仍按 micro-batch 计算；
- 每次 optimizer update 之间耗时更长；
- learning-rate schedule 应按 optimizer step，而不是 dataloader iteration 推进；
- 日志里的“step”必须说清是 micro-step 还是 update-step。

diffusion U-Net 常用 GroupNorm，减轻了小 micro-batch 的 BatchNorm 问题，但步数口径仍不能混。

### 台阶 8：多卡 DDP 的核心不是“复制模型”，而是同步梯度

DistributedDataParallel（DDP）中，每张 GPU 各有一份模型副本，各自处理不同数据。反向传播时，对同名参数的梯度做 all-reduce，得到跨卡平均梯度，再让各卡执行相同更新。

要让它真的等价于单卡大 batch，需要满足：

- 用 `DistributedSampler`，避免每张卡重复读同一批样本；
- 每个 epoch 调用 `sampler.set_epoch(epoch)`，保证各卡 shuffle 一致且会变化；
- 只让 rank 0 写 checkpoint 和主日志；
- 随机种子既要可复现，又要让不同 rank 的增强和噪声不完全相同；
- accumulation 的非更新步使用 `no_sync()`，否则每个 micro-step 都通信；
- resume 时恢复 model、EMA、optimizer、scheduler、scaler 和 global step。

多卡最隐蔽的错误，不是程序崩溃，而是“看起来在训练，实际每卡数据重复”“学习率按错 batch 缩放”或“验证时各 rank 重复统计同一批样本”。

## 实现：一条最小但契约完整的训练骨架

下面代码展示 cosine schedule、`v-prediction`、AMP、gradient clipping 和 EMA 的核心关系。为突出训练契约，省略了数据集和 DDP 初始化。

```python
import copy
import math
import torch
import torch.nn.functional as F


def cosine_alpha_bar(num_steps, s=0.008, device="cuda"):
    t = torch.linspace(0, num_steps, num_steps + 1, device=device, dtype=torch.float64)
    f = torch.cos(((t / num_steps + s) / (1 + s)) * math.pi / 2).square()
    alpha_bar = (f / f[0]).clamp(min=1e-8, max=1.0)
    return alpha_bar.float()


@torch.no_grad()
def update_ema(ema_model, model, decay=0.9999):
    online = model.module if hasattr(model, "module") else model
    for p_ema, p in zip(ema_model.parameters(), online.parameters()):
        p_ema.mul_(decay).add_(p.detach().float(), alpha=1.0 - decay)


def train_step(model, ema_model, optimizer, scaler, x0, cond, alpha_bar):
    batch = x0.shape[0]
    t = torch.randint(0, len(alpha_bar), (batch,), device=x0.device)

    a = alpha_bar[t].sqrt().view(batch, 1, 1, 1)
    sigma = (1.0 - alpha_bar[t]).sqrt().view(batch, 1, 1, 1)
    noise = torch.randn_like(x0)
    xt = a * x0 + sigma * noise

    # v = alpha * epsilon - sigma * x0
    target_v = a * noise - sigma * x0

    optimizer.zero_grad(set_to_none=True)
    with torch.autocast(device_type="cuda", dtype=torch.float16):
        pred_v = model(xt, t, cond)
        loss = F.mse_loss(pred_v.float(), target_v.float())

    scaler.scale(loss).backward()
    scaler.unscale_(optimizer)
    torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
    scaler.step(optimizer)
    scaler.update()
    update_ema(ema_model, model)
    return loss.detach()


# 初始化示意：EMA 保留 float32，并且不求梯度
# ema_model = copy.deepcopy(model).float().eval()
# for p in ema_model.parameters():
#     p.requires_grad_(False)
```

一条可靠训练线还应在 checkpoint 里保存：

```python
checkpoint = {
    "model": model.state_dict(),
    "ema": ema_model.state_dict(),
    "optimizer": optimizer.state_dict(),
    "scaler": scaler.state_dict(),
    "global_step": global_step,
    "prediction_type": "v_prediction",
    "schedule_config": schedule_config,
}
```

最后两项不是装饰。没有 prediction type 和 schedule 配置，即使权重文件还在，也可能无法按原训练契约正确采样。

## 🔗 与我 CVPR 工作的连接

在偏振表征生成或 reliability-guided dense prediction 中，训练工程本身就会影响论文结论。

建议把以下信息写进配置、日志和 checkpoint：

- 每个模态的归一化范围与逆变换；
- AoLP 是否用 `sin(2θ), cos(2θ)` 表示；
- 低 DoLP 区域是否用 reliability mask 降权；
- prediction type 是 `epsilon`、`x0` 还是 `v`；
- noise schedule 和 timestep sampling；
- EMA decay，验证是否使用 EMA；
- effective batch、world size、accumulation 和 learning rate；
- AMP dtype 与出现 `NaN/inf` 时的处理。

尤其要防一个论文级混淆：如果 baseline 用 EMA 权重评估，而你的方法用最后一步 online 权重评估，二者不是同一训练口径；如果不同模态归一化不同，统一 MSE 也不代表统一难度。

一个值得做的小实验是按 SNR 分桶统计验证误差：

```text
low SNR / mid SNR / high SNR
分别比较 RGB fidelity、DoLP error、AoLP circular error、normal angular error
```

它能回答“模型到底在哪个噪声区间学不好”，比只看一个平均 diffusion loss 更接近 reliability 机制的核心。

## 一句话总结

> Diffusion 训练稳定，不靠某一个神奇超参数，而靠一组一致的契约：noise schedule 安排 SNR 课程，prediction target 定义网络输出，EMA 平滑评估权重，mixed precision 管住数值范围，DDP 和 accumulation 管住有效 batch。任何一处口径错位，都可能让“模型问题”其实变成工程问题。

## 🔧 配菜·面试考点

**Q：`ε-prediction` 和 `v-prediction` 有什么区别？为什么训练和 sampler 必须匹配？**

**A：** `ε-prediction` 直接预测加到 `x_0` 上的噪声；`v-prediction` 预测 `v = αε - σx_0`，它把数据与噪声方向按 timestep 组合，在不同 SNR 下通常更平衡。由 `x_t` 和 `v` 可恢复 `x_0 = αx_t - σv`、`ε = σx_t + αv`。sampler 必须知道模型输出属于哪种参数化，才能把输出换算成正确的反向更新；若把 `v` 当 `ε`，每一步的 `x_0` 和方向估计都会错。

追问预备：EMA 为什么不参与反向传播；AMP 为什么要 unscale 后再 gradient clipping；DDP 下 effective batch 怎样计算。
